# jPipe VS Code Extension — CLAUDE.md

## Project Overview

This is a VS Code extension and Language Server for **jPipe**, a DSL for justification modeling (used in software certification contexts at McMaster University / McSCert). Users write `.jd` files describing justification structures; the extension provides syntax highlighting, validation, completions, and diagram preview/export.

**Stack:** TypeScript, Langium 4.2 (language framework), Vitest (tests), esbuild (extension bundler), vsce (publishing).

## Language Grammar Summary

The grammar (`packages/language/src/jpipe.langium`) defines:

| Concept | Keyword | Notes |
|---|---|---|
| **Justification** | `justification Name implements Template?` | Concrete model |
| **Template** | `template Name implements Template?` | Abstract/reusable structure |
| **Evidence** | `evidence name is "label"` | Leaf node |
| **Strategy** | `strategy name is "label"` | Must be supported by ≥1 evidence/sub-conclusion |
| **SubConclusion** | `sub-conclusion name is "label"` | Intermediate conclusion |
| **Conclusion** | `conclusion name is "label"` | Root; must be supported by ≥1 strategy |
| **AbstractSupport** | `@support name is "label"` | Like abstract method; only in templates |
| **Relation** | `from supports to` | Directed support edge |
| **Load** | `load "path/to/file.jd"` | Import another `.jd` file |
| **Composition** | `composition { ... }` | Assembles/refines models with operators |

Inheritance: both justifications and templates may `implements` a parent template. `getAllElements` traverses the chain.

---

## Key Design Decisions

### @support = abstract elements
`@support` in a template is like an abstract method: justifications implementing the template **must** override every `@support` element with an `evidence` or `sub-conclusion` (type-checked). Regular (non-@support) elements are optional overrides.

### Import resolution
`JpipeImportService` does a BFS over `load` edges. Direct imports must appear in the root document; transitive imports are followed freely. Resolution is path-based (no workspace index for imports).

### Completion scoping
The `JpipeCompletionProvider` deliberately never falls through to Langium's global workspace index for `JustificationElement` or `Template` references — it restricts candidates to local + explicitly loaded files only. The one exception is `implements` in the `parentTemplateCandidateDescriptions` method, which does consult the workspace index and can auto-insert a `load` statement.

### Image generation
`ImageGenerator` shells out to an external **jPipe CLI** (default: `jpipe` on PATH) or a **JAR file**. The extension does not bundle the compiler. Configure via:
- `jpipe.cliPath` (takes priority)
- `jpipe.jarFile` + `jpipe.setJavaVersion`

Exit codes: `1` = model has errors, `42` = compiler crashed.

## Build & Dev Workflow

```bash
# First time / dependency install
npm install
npm install -g @vscode/vsce          # for packaging

# Regenerate AST/grammar from jpipe.langium (run after any grammar change)
npm run langium:generate

# Full build
npm run build

# Watch mode
npm run watch

# Run in VS Code (launches Extension Development Host)
# → press F5 in VS Code

# Tests (Vitest, language package only)
npm test

# Package the extension as a .vsix
cd packages/extension
vsce package -o jpipe-vscode.vsix

# Install locally
code --install-extension jpipe-vscode.vsix

# Publish to marketplace
vsce publish
```

**Important:** The files under `packages/language/src/generated/` are generated artifacts. Always re-run `npm run langium:generate` after editing `jpipe.langium`.

## Testing

Tests live in `packages/language/test/`. They use Vitest with Langium's `parseHelper`.

- `parsing.test.ts` — parse smoke tests (currently skipped/stub, needs real jPipe examples)
- `linking.test.ts` — cross-reference resolution
- `validating.test.ts` — validator rule checks

The parsing test's `beforeAll` wires up services with `EmptyFileSystem` (no real disk). For import-heavy tests, prefer constructing multi-document scenarios manually via `JpipeImportService.parseDocumentFromPath`.

## Code Conventions

- All source is TypeScript (ES modules, `"type": "module"`), targeting Node 22.
- The extension bundle is CJS (esbuild converts it); the language server stays ESM.
- Langium services are wired via constructor injection; access shared services through the injected `JpipeServices` handle, not via global imports.
- AST node type guards (`isJustification`, `isTemplate`, etc.) come from `./generated/ast.js` — use them instead of `instanceof` or `.$type` comparisons.
- Prefer `getAllElements` (includes inherited chain) over `getLocalElements` (only directly declared) unless you explicitly want only local elements.
